# RACE : RunAndCatchEnding
### 1. 게임 소개
    - **달려서 끝을 붙잡으세요!
    RACE : Run And Catch Ending은 플레이어가 올바른 포즈를 취해 상대보다 먼저 결승선에 도달하면 승리하는 레이싱 게임입니다!**
--- 
### 2. 시놉시스

    5월의 어느 목요일, 컴퓨터 비전 수업을 듣는 두 학생. 잠에서 깨어났을 땐, 16시 30분. 지각은 고사하고 결석 할 위기! 다행히 둘의 친구 중 한 명이 스마트 출석을 했지만 잘 안된다고 말해 한 자리의 출석이 남은 상황. 먼저 도착하는 사람이 출석을 쟁취할 수 있다!

    (해당 시놉시스는 허구이며 실화가 아닙니다.)

### 3. 게임 방법
    - 실행 방법
        1. 레포지토리에서 dist에 있는 Race 폴더를 다운로드 한다.
        2. Race 폴더 내에 “Race.exe” 파일을 실행한다.
    - 플레이 방법
        1. 시작화면에서 Start를 누릅니다.
        2. 로딩 후, 카운트다운이 끝나면 시작!
        3. 왼쪽에 있는 사람이 플레이어 1, 오른쪽에 있는 사람이 플레이어 2!
        4. 화면 왼쪽 위에 나오는 포즈를 따라하면 테두리가 초록색으로 빛나며 속도가 빨라집니다.
        5. 포즈를 일정 시간 내에 따라하지 못하면 속도가 느려집니다.
        6. 상대보다 더 먼저 결승선에 도달하면 승리!
    - 팁
        - 카메라에서 충분히 떨어져 얼굴, 팔, 골반까지 다 나오도록 서주세요! 팔은 뻗었을 때 카메라 에 다 잡혀야 합니다.
        - 올바른 포즈를 취하고 잠깐 멈춰주세요. 그래야 성공으로 인식합니다.
        - 포즈를 인식하지 못한다면 팔의 각도를 달리하며 확인해주세요. 각도 조건을 못 맞췄을 가능성이 큽니다.
        - 팔이 중앙 경계를 넘어도 어느 정도 괜찮습니다.
    - 게임 화면 설명

### 4. 게임 시스템 및 구현 단계에서 겪었던 문제
    - 포즈 룰에서 겪었던 문제 - 기준을 느슨하게? 엄밀하게?
        - 포즈를 인식하는 문제는 게임의 재미에 큰 영향을 줄 수 있기에 주의 깊게 결정해야 했습니다. 너무 기준을 엄밀하게 잡는다면 플레이어가 아무리 포즈를 취해도 성공하지 못하니 지루하리라 생각해 가능하면 포즈의 조건은 빡빡하게 잡지 않았습니다. 애초에 사람의 부위를 인식하는 mediapipe가 정확히 인식을 못하는 경우도 있을 수 있기에 느슨한 기준을 잡아야 한다고 생각했습니다. 그래서 기준은 주로 손목, 팔꿈치, 어깨의 각도를 확인하거나 신체 부위의 상대적인 위치(가령 손목이 어깨 라인보다 위에 있거나, 팔꿈치가 손목보다 아래에 있는 등)로 확인해 엄밀하게 정하지는 않았습니다. 추가로 허용하는 오차 각, 거리를 설정해 무조건 요구하는 각도나 상대적인 위치를 맞추지 않고 살짝 벗어나도 포즈를 허용했습니다.
        - 포즈 기준을 느슨하게 해서 생기는 문제도 있었습니다. no의 포즈와 죠죠 서기1 포즈는 이미지로 봤을 때는 확연히 다르나, 팔의 각도나 위치 기준으로 봤을 때는 비슷한 포즈였기에 요구한 포즈가 no임에도 다른 포즈를 취했을 때 인식했습니다. 그래서 죠죠 서기1의 포즈를 내린 팔의 각도를 더 크게 하는 식으로 변형하여 오인식을 피하고자 했습니다.
    - 포즈의 연속 인식 문제
        - 처음 코드에서는 한 프레임에 포즈가 조건과 일치하면 성공하고, 성공한 다음 프레임에도 포즈가 일치하면 성공할 수 있었습니다. 그러다 보니 포즈를 바꾸는 도중에도 포즈 조건과 비슷해서 성공이라 인식해버리거나 연속적으로 2~3개의 포즈가 성공으로 인식하는 경우가 있었습니다.
        - 이런 문제를 해결하기 위해 한 프레임동안 포즈가 성공하는 것이 아닌 0.3초 동안 포즈가 조건과 일치해야 성공으로 판단하는 것으로 기준을 바꾸고, 한번 성공하고 1초 동안은 포즈를 성공해도 실패라고 인식하도록 바꿨습니다.
    - 화면 구성의 변화
        - 처음에는 화면 구성은 이러했습니다.
        - 화면은 절반으로 나눠 각 플레이어가 화면을 반 씩 가져가도록 하고, 그 화면에서는 카메라 화면, 플레이어가 취해야 할 포즈 이미지, 달리고 있는 게임 캐릭터와 플레이어 속도, 남은 거리를 표시하려고 했습니다.
        - 그러나 테스트를 했을 때, 사람의 상반식이 화면에 들어가려면 생각보다 노트북과의 거리가 멀어야 했고  그러면 화면에 표시되는 것들이 잘 보이지 않았습니다.
        - 저는 공간의 부족을 느껴 나눠진 화면에서 각 파트를 또 나누지 않고, 나눠진 화면에 카메라 이미지가 꽉 채우도록 했습니다. 그 이미지 위에 목표 포즈, 속도, 플레이어의 위치를 나타내는 직선을 반투명으로 표시해 지금 내가 취하고 있는 포즈와 현재 상태를 확인할 수 있도록 만들었습니다.

### 5. 코드 전체적인 구조 설명
    - 클래스 간 연결 관계 및 클래스 설명

        ```text
        GameManager
        ├─ StartScreen
        ├─ Renderer
        ├─ PoseConnector
        │  ├─ PoseDetector
        │  │  ├─ Camera
        │  │  └─ LandmarkSmoother
        │  └─ PoseLibrary
        │     ├─ PoseRule
        │     └─ PoseTolerance
        └─ PlayerState
        ```

        - GameManager : 게임 전체를 관리하는 클래스입니다. 하위 클래스를 통해 시작 화면, 로딩, 게임 화면, 종료 화면과 게임의 루프를 책임집니다.
            - StartScreen : 게임을 실행할 때 처음 보이는 화면을 띄우고 시작 버튼과 종료 버튼으로 GameManager가 다음 해야 할 동작을 알립니다.
            - Renderer : 게임 화면을 띄우고 이펙트 및 사운드를 처리하는 클래스입니다. 이미지 프레임을 받아 프레임 위에 플레이어 UI를 그리고 플레이어 상태를 받아서 이펙트나 사운드를 보냅니다.
            - PoseConnector : PoseLibrary에서 포즈 기준을 받고 PoseDetector에서 플레이어의 랜드마크(각 팔의 팔꿈치, 어깨, 손목, 코, 귀, 골반)을 받아 해당 포즈를 취하고 있는지 확인하여 성공 여부를 보내는 클래스입니다.
                - PoseLibrary : PoseConnector에게 호출 받아 무작위로 포즈 기준을 보내고 포즈 기준에 대한 정보를 가지는 클래스입니다. PoseRule은 각 포즈의 규칙, 이름, 별칭, id에 대한 데이터 클래스고 PoseTolerance는 얼마나 오차를 허용할지 나타내는 데이터 클래스입니다.
                - PoseDetector : Camera에서 찍힌 프레임을 받아 mediapipe를 통해 사람의 신체 부위인 랜드마크를 뽑는 클래스입니다. 해당 랜드마크로 프레임에서 사람의 신체 부위에 점을 찍고 선을 그리는 역할도 담당합니다. Camera는 opencv를 통해 카메라로 플레이어의 모습을 찍어 PoseDetector에게 넘깁니다. LandmarkSmoother는 매 프레임마다 그려진 점과 선이 부드럽게 움직이도록 하는 클래스입니다.
            - PlayerState : 플레이어의 현재 속도, 현재 위치, 남은 위치 등 플레이어의 상태를 확인하고 업데이트하는 클래스입니다. 포즈 성공 여부에 따라 속도를 증감하는 역할과 완주를 확인하는 역할을 합니다.
    - 전체 실행 흐름

        ```text
        GameManager.launch()
        -> StartScreen.run()
        -> GameManager.start()
          -> Renderer.show_loading()
          -> GameManager._setup_game()
          -> GameManager.prepare_game()
          -> Renderer.show_countdown()
        -> GameManager.run()
        ```

    - 각 함수 내 실행 순서 설명
        - StartScreen.run() : 처음 시작 화면을 띄우고 배경 음악을 재생하며 버튼의 입출력을 받아 게임 매니저에게 start 버튼이 눌렸는지, exit 버튼이 눌렸는지 리턴합니다. 이를 받은 GameManager는 시작 버튼이 눌렸을 때 start()를 호출합니다.
        - GameManager.start() : 게임을 시작할 준비를 완료하고 run()을 호출해 게임 메인 루프를 시작합니다. 먼저 Renderer를 생성합니다.
            - Rendere.show_loading() : 게임이 준비될 동안 보여줄 로딩 화면을 띄우고 로딩 사운드를 출력합니다.
            - GameManager._setup_game() : 게임 메인 루프에 핵심이 되는 PoseConnector와 PlayerState를 생성합니다. 이때 둘의 하위 클래스도 같이 생성됩니다.
            - GameManager.prepare_game() : PoseConnector와 PlayerState의 prepare_game()을 호출해 두 클래스가 게임을 준비하게 합니다. 이때 카메라를 키고 한 프레임을 찍어 리턴합니다.
            - Rendere.show_countdown() : 이전 함수에서 찍었던 프레임 위에 “Ready? 3 2 1 Go!” 텍스트를 띄웁니다.
    - GameManager.run() 실행 흐름

        ```text
        GameManager.run()
        -> while True 반복
          -> PoseConnector.update()
          -> 각 PlayerState.update()
          -> Renderer.make_render_data()
          -> Renderer.render()
          -> GameManager._get_winner_id()
          -> 만약 결승선에 누가 도달했다면 - GameManager.finish() + break
          -> 만약 “q”를 눌렀다면 - Renderer.should_quit() + break
        -> finally
          -> GameManager.close(quit_pygame=False)
        ```

        - PoseConnector.update() : Camera 클래스로 이미지를 PoseDetector가 받아 랜드마크를 뽑아 PoseConnector에게 넘깁니다. 이를 받은 PoseConnector는 현재 가지고 있는 두 플레이어의 PoseRule의 check 함수로 포즈가 일치하는 지 확인하여 성공 여부를 결정합니다. 성공 여부와 PoseDetector가 만든 프레임을 리턴합니다.
        - PlayerState.update() : 두 플레이어의 상태를 업데이트 합니다. 이전에 PoseConnector가 리턴한 성공 여부를 가지고 속도 증감을 하고 정해진 속도로 현재 위치와 남은 거리를 계산합니다. 마지막으로 계산된 결과로 결승선에 도달하는 지 확인합니다. 현재 속도, 현재 위치, 남은 거리, 결승선 도달 여부를 리턴합니다.
        - Renderer.make_render_data() : PlayerState와 PoseConnector가 리턴한 결과로 렌더러를 위한 데이터를 만듭니다.
        - Renderer.render() : 위 함수에서 만든 데이터를 받아 성공 여부에 따라 이펙트, 사운드를 출력하고 PoseConnector가 전달한 프레임과 PlayerState로 받은 상태로 각 플레이어의 UI를 그립니다.
        - GameManager._get_winner_id() : PlayerState가 전달한 결승선 도달 여부를 보고 도달한 플레이어가 있다면 그 플레이어의 id를 리턴합니다.
        - 만약 결승선에 누군가 도달했다면 GameManager.finish()를 호출하고 renderer가 show_finish()로 종료 화면을 송출 후 시작 화면으로 돌아갑니다.
        - 만약 “q”를 눌러 게임을 종료하면 Rendner.should_quit()으로 시작 화면으로 돌아갑니다.
        - finish()나 should_quit()으로 게임이 종료되면 close() 함수가 실행되어 다시 게임을 할 수 있게 정리를 시작합니다.

### 6. 바이브 코딩하면서 느낀 점
    - 매우 빠른 구현 + 능력 외에 것도 구현 가능
        - 저번 오픈 소스 소프트웨어 수업의 프로젝트에서는 주로 손 코딩을 했고 이미 만들어낸 카드 설명을 코드로 옮기는 단순 작업에는 ai로 구현했었습니다. 이때 기획을 제외하고 프로그래밍한 시간이 2주 좀 넘게 되었습니다. 이번엔 코드 구조를 제가 생각하고 코딩하는 건 ai에게 맡겼는데 거의 4일 만에 모든 코드를 다 짤 수 있었습니다. 원래 ai를 사용하지 않았다면 최소 1주일 반에 2주 정도는 걸렸을 작업이 4일 안에 해결할 수 있었다는 점에서 바이브 코딩의 속도를 체감할 수 있었습니다.
        - 사실 파이썬은 아직도 미숙하고, 처음보는 mediapipe 라이브러리나 지난 오소스 수업에서 잠깐 써본 pygame 라이브러리는 잘 알지도 못했습니다. 그럼에도 ai가 이런 라이브러리와 파이썬을 잘 알고 있어 제 능력 외에 것도 빠르게 구현이 가능했습니다.
    - 블랙박스의 공포, 생각보다 멍청한 ai?
        - 그러나 문제도 있었는데 그 중 첫 번째가 블랙박스였습니다. ai에게 코딩을 맡기다 보니 저는 ai가 써준 코드를 이해하는데 시간을 많이 소모하게 되었습니다. ai는 5분도 안 되서 코드를 다 짜는데 저는 그 코드를 이해하는 데 10분이 걸리는, 즉 역으로 병목 현상이 발생하게 되었습니다. 또한, 이렇게 시간을 써서 이해하려고 해도 코드를 잘 이해하기가 어려웠습니다. 그러다 보니 이 코드가 구조가 잘 짜여있는지, 비효율적인 부분이 있는지 확인할 수 없었습니다. 코드를 잘 파악하지 못하다 보니, 실행할 때마다 버그가 발생하지 않기를 빌어야 하고 버그가 나도 제가 처리하기 보단 ai가 찾아주길 바래야 합니다. 이런 점에서 코드는 프로그래머가 책임져야 하는데 바이브 코딩은 그 책임을 보장해주기 어렵다고 느꼈습니다.
        - 두 번째 문제는 생각보다 ai가 멍청하다는 것 입니다. 새로운 포즈 기준을 추가할 때, 충분히 기존에 있던 보조 함수로 해결할 수 있던 걸 조금 다르다고 완전히 새 함수를 ai가 구현했었습니다. 그리고 자그마한 것을 구현할 때 최대한 많은 보조 함수를 만들어서 구현하려 한다고 느꼈습니다. 작은 코드 라인 하나하나는 ai가 더 효율적으로 짤 수 있어도, 전체적인 구조나 코드 스타일에서는 문제가 있다고 생각했고, 코드 구조, 스타일, 자료구조는 프로그래머가 배워서 조정해야 한다고 느꼈습니다.
    - 어떻게 하면 이런 ai를 더 잘 쓸 수 있을까?
        - 가장 먼저 든 생각은 ai의 코드를 책임질 수 있게 어떤 의도와 맥락으로 코드를 짜고 그 코드가 어떤 역할을 하는지 파악해야 한다는 것입니다. 이를 그냥 프로그래머가 파악하려고 한다면 시간이 많이 걸리기에, ai를 조정해서 자신의 코드에 대해 어떤 의도를 가지고 이 함수를 만들었는지 설명하도록 만들어야 합니다. 또한, 어떤 코드 구조와 스타일로 짤지 미리 결정해두고 이를 검사하는 ai를 두어 코드 짜는 ai가 그 규칙을 잘 따르는지 검사하도록 하면 시간을 절약하면서 더 효율적으로 바이브 코딩을 할 수 있다고 생각했습니다.
        - 차후 바이브 코딩을 하며 어떤 식으로 하는 게 코드를 잘 파악하면서도 시간 효율적으로 할 수 있는지 확인해볼 예정입니다.
